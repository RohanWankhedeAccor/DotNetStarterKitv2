## Naming Conventions

### Backend (C#)

#### Classes & Namespaces
- **PascalCase** for all types: `User`, `CreateUserCommand`, `IApplicationDbContext`
- **Feature folder names**: lowercase plural: `Features/Users/`, `Features/Products/`
- **File names**: match class name: `CreateUserCommand.cs` contains `CreateUserCommand` class
- **Namespace convention**: `Api.Endpoints.Users`, `Application.Features.Users.Commands`

#### Methods & Properties
- **PascalCase** for public methods/properties: `GetUserById()`, `FullName`
- **camelCase** for parameters: `pageNumber`, `cancellationToken`
- **Prefixes for boolean methods**: `IsActive`, `HasPermission()`, `CanDelete()`

#### Commands, Queries, DTOs
- **Commands**: `CreateUserCommand`, `UpdateUserCommand`, `DeleteUserCommand`
- **Queries**: `GetUserByIdQuery`, `GetUsersQuery`
- **DTOs**: `CreateUserDto`, `UserDto`, `UserResponseDto`
- **Handlers**: `CreateUserCommandHandler`, `GetUserByIdQueryHandler`
- **Validators**: `CreateUserValidator`
- **Profiles (AutoMapper)**: `CreateUserCommandProfile`

#### Database/EF Core
- **Entity names**: singular, PascalCase: `User`, `Role`, `Product`
- **Table names**: plural (via EF config): `Users`, `Roles`, `Products`
- **Foreign key properties**: `UserId`, `RoleId` (PascalCase)
- **Navigation properties**: singular or plural based on relationship: `Role`, `UserRoles`
- **Configuration classes**: `UserConfiguration`, `ProductConfiguration`

### Frontend (TypeScript/React)

#### Files & Folders
- **Component files**: PascalCase: `UsersList.tsx`, `CreateUserForm.tsx`, `UserCard.tsx`
- **Hook files**: camelCase: `useUsers.ts`, `useCreateUser.ts`, `useAuth.ts`
- **Type files**: lowercase: `types.ts`, `index.ts`
- **Test files**: `*.test.ts` or `*.test.tsx`
- **Folder names**: lowercase plural: `features/users/`, `components/`, `hooks/`

#### Components & Functions
- **Component names**: PascalCase: `UsersList`, `CreateUserForm`, `ErrorBoundary`
- **Hook names**: camelCase starting with `use`: `useUsers`, `useCreateUser`, `useFetch`
- **Utility functions**: camelCase: `formatDate()`, `validateEmail()`, `parseError()`
- **Constants**: UPPER_SNAKE_CASE: `API_BASE_URL`, `PAGE_SIZE`, `QUERY_KEYS`

#### Types & Interfaces
- **Interface names**: PascalCase: `User`, `UserDto`, `CreateUserRequest`
- **Enum names**: PascalCase: `UserStatus`, `ProjectStatus`
- **Type names**: PascalCase: `type UserResponse = { ... }`

#### Variables & States
- **Local state**: camelCase: `const [users, setUsers] = useState(...)`
- **Query keys**: nested array: `['users', { pageNumber: 1 }]`
- **Redux state keys**: camelCase: `authSlice`, `userSlice`

## Commit Message Conventions

Follow **Conventional Commits** style for consistency and automated changelog generation.

### Format
```
<type>(<scope>): <subject>

<body>

<footer>
```

### Types
- **feat**: New feature
- **fix**: Bug fix
- **refactor**: Code restructuring (no feature change)
- **test**: Adding or updating tests
- **docs**: Documentation updates
- **chore**: Build, dependencies, tooling
- **perf**: Performance improvements
- **style**: Code style (formatting, missing semicolons, etc.)

### Scope (Optional but Recommended)
Scope of the change: `users`, `products`, `auth`, `api`, `frontend`, `db`

### Subject
- Imperative mood: "add" not "added" or "adds"
- No period at end
- Max 50 characters
- Lowercase start

### Body (Optional)
- Wrapped at 72 characters
- Explain **what** and **why**, not how
- Use bullet points for lists

### Examples

**Good:**
```
feat(users): add user creation endpoint

- Implement POST /api/v1/users handler
- Add CreateUserValidator with email uniqueness check
- Create UserDto and CreateUserDto
- Add integration test for user creation

Closes #123
```

**Good:**
```
fix(products): resolve null reference in ProductsList

Previously, ProductsList crashed when products list was empty.
Now it displays a user-friendly "No products found" message
with a link to create the first product.
```

**Good:**
```
refactor(api): extract health check to separate endpoint
```

**Bad:**
```
fix bug
updated stuff
added new things
```

## Pull Request Checklist

Before creating a PR, verify:

### Code Quality
- ✅ Code follows naming conventions above
- ✅ No `TODO`, `FIXME`, `HACK` comments left uncommitted
- ✅ No debug `Console.WriteLine()` or `console.log()`
- ✅ No unused imports or variables
- ✅ No `var` in C# (use explicit types)
- ✅ All public methods have XML documentation (`/// <summary>`)

### Backend
- ✅ `dotnet build` passes (zero warnings)
- ✅ `dotnet test` passes (all tests green)
- ✅ All new handlers have validators
- ✅ All database reads use `AsNoTracking()`
- ✅ All DTOs used at API boundaries (not entities)
- ✅ No `Task.Result` or `.Wait()` (async all the way)
- ✅ New migrations committed (if touching database)
- ✅ Connection string not hardcoded (uses `appsettings.json`)

### Frontend
- ✅ `npm run lint` passes
- ✅ `npm run type-check` passes (zero TS errors)
- ✅ `npm run test` passes (all tests green)
- ✅ No hardcoded API URLs (uses environment variables)
- ✅ New components wrapped in `<ErrorBoundary>`
- ✅ Loading states shown (skeleton, spinner, etc.)
- ✅ Error states handled (toast or error message)
- ✅ Responsive design verified (mobile, tablet, desktop)

### Documentation
- ✅ README updated if behavior changed
- ✅ API endpoint documented in swagger/openapi
- ✅ New environment variables documented in `.env.example`
- ✅ Breaking changes highlighted in PR description

### Testing
- ✅ New features have unit or integration tests
- ✅ Coverage target: 80%+ for backend, 70%+ for frontend
- ✅ Edge cases tested (empty list, null values, errors)

## Code Review Standards

### What We Look For

#### Correctness
- Does the code solve the stated problem?
- Are there any obvious bugs or edge cases missed?
- Could this break in production?

#### Architecture
- Does it follow Clean Architecture (Domain ← App ← Infrastructure ← API)?
- Are dependencies correct (nothing points upward)?
- Is it in the right folder/layer?

#### Maintainability
- Is the code easy to understand?
- Could someone new pick it up in 5 minutes?
- Are there enough (but not too many) comments?

#### Testing
- Are there tests? Do they cover the happy path AND error cases?
- Would you catch a regression with these tests?

#### Performance
- Any N+1 queries? Missing indexes?
- Any unnecessary re-renders in React?
- Any obvious memory leaks?

#### Security
- Are secrets hardcoded anywhere?
- Is user input validated?
- Are there CORS/CSRF issues?

### Reviewer Responsibilities
- Be respectful and constructive
- Ask questions instead of making demands
- Approve quickly if code is good
- Block only on serious issues (bugs, architecture, security)
- Request changes for style/naming if it's a pattern we follow

### Author Responsibilities
- Respond to all feedback
- Resolve conversations after making changes
- Don't merge without at least one approval
- Rerun tests locally before merging

## Testing Standards

### Backend (xUnit + NSubstitute + FluentAssertions)

#### Unit Tests
- Test a single class/method in isolation
- Mock dependencies (use NSubstitute)
- Name pattern: `MethodName_Scenario_ExpectedResult`
- Example: `CreateUserCommand_WithValidEmail_ShouldSucceed()`

```csharp
[Fact]
public async Task CreateUser_WithValidEmail_ShouldSucceed()
{
    // Arrange
    var command = new CreateUserCommand { Email = "test@example.com" };
    var handler = new CreateUserCommandHandler(mockDb, mockValidator);

    // Act
    var result = await handler.Handle(command, CancellationToken.None);

    // Assert
    result.Should().NotBeNull();
    result.Email.Should().Be("test@example.com");
}
```

#### Integration Tests
- Test multiple layers together (no mocks)
- Use real `ApplicationDbContext` with in-memory SQLite
- Name pattern: same as unit tests
- Cover happy path + error scenarios

```csharp
[Fact]
public async Task CreateUser_ShouldPersistToDatabase()
{
    // Arrange - use real context
    var context = new ApplicationDbContext(dbOptions);
    var handler = new CreateUserCommandHandler(context, validator);
    var command = new CreateUserCommand { Email = "test@example.com" };

    // Act
    await handler.Handle(command, CancellationToken.None);
    var user = await context.Users.FirstOrDefaultAsync(u => u.Email == "test@example.com");

    // Assert
    user.Should().NotBeNull();
}
```

### Frontend (Vitest + React Testing Library + MSW)

#### Component Tests
- Render component with necessary providers
- Use `screen.getByRole()` (accessibility-first)
- Name pattern: describe what user sees
- Example: `should display user list with pagination`

```typescript
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import { UsersList } from './UsersList'

describe('UsersList', () => {
  it('should display list of users', () => {
    render(<UsersList users={[{ id: 1, email: 'test@example.com' }]} />)
    expect(screen.getByText('test@example.com')).toBeInTheDocument()
  })

  it('should call onUserClick when user is clicked', async () => {
    const mockClick = vi.fn()
    render(<UsersList users={[...]} onUserClick={mockClick} />)
    await userEvent.click(screen.getByText('test@example.com'))
    expect(mockClick).toHaveBeenCalled()
  })
})
```

#### Hook Tests
- Use `renderHook()` from `@testing-library/react`
- Mock API responses with MSW
- Test both success and error scenarios

```typescript
import { renderHook, waitFor } from '@testing-library/react'
import { useUsers } from './useUsers'
import { server } from '../test/msw'

describe('useUsers', () => {
  it('should fetch users on mount', async () => {
    const { result } = renderHook(() => useUsers())
    await waitFor(() => expect(result.current.data).toBeDefined())
    expect(result.current.data).toHaveLength(2)
  })

  it('should handle API error', async () => {
    server.use(http.get('/api/v1/users', () => HttpResponse.error()))
    const { result } = renderHook(() => useUsers())
    await waitFor(() => expect(result.current.isError).toBe(true))
  })
})
```

### Coverage Targets
- Backend: 80%+ (handlers, validators, domain logic)
- Frontend: 70%+ (components, hooks, utils)
- **Skip testing**: DTOs, AutoMapper profiles, Redux reducers (simple logic), DI configuration

## Common Pitfalls to Avoid

### Backend
- ❌ Returning `IQueryable` from repositories (return materialized lists)
- ❌ Using `Task.Result` or `.Wait()` in async code
- ❌ Forgetting `AsNoTracking()` on read-only queries
- ❌ Returning entire entities from handlers (use DTOs)
- ❌ Business logic in controllers (put it in handlers/services)
- ❌ Hardcoding connection strings (use configuration)
- ❌ Missing validation on command handlers
- ❌ Not handling soft-delete globally (rely on query filter)

### Frontend
- ❌ Calling `useQuery`/`useMutation` conditionally
- ❌ Storing server state in Redux (use TanStack Query)
- ❌ Making API calls in useEffect without dependency array
- ❌ Not mocking API in tests (use MSW)
- ❌ Hardcoded API URLs (use environment variables)
- ❌ Missing error boundaries around feature components
- ❌ Not handling loading/error states in UI
- ❌ Re-fetching data on every render

## Git Workflow

### Branch Naming
- Feature: `feat/user-authentication`
- Bug fix: `fix/invalid-email-validation`
- Refactor: `refactor/api-error-handling`
- Docs: `docs/api-setup-guide`

### Before Pushing
```bash
# Backend
dotnet build
dotnet test --no-build --verbosity normal

# Frontend (from src/Web)
npm run lint
npm run type-check
npm run test

# Then commit and push
git push origin branch-name
```

### Merge Strategy
- Use **Squash and Merge** for single-commit features
- Use **Create a Merge Commit** for multi-commit refactors
- **Always delete remote branch after merge**
